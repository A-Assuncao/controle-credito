# ADR-0015: SLOs e operação inicial

- **Status:** Aceito
- **Data:** 2026-06-20

## Contexto

Seção 23.3 do plano sugere SLOs iniciais. Precisam ser formalizados como compromisso e base para runbooks.

## Decisão

**SLOs iniciais (v1):**

| Serviço                 | SLO                            | Métrica                            | Janela     |
| ----------------------- | ------------------------------ | ---------------------------------- | ---------- |
| API crítica             | Disponibilidade ≥ 99.9% mensal | Taxa de erro < 0.1% (5xx)          | mensal     |
| API crítica             | Latência p95 ≤ 400ms           | `http_server_duration`             | mensal     |
| API crítica             | Latência p99 ≤ 1s              | `http_server_duration`             | mensal     |
| Notificações e-mail     | Processamento p95 ≤ 60s        | `notification_processing_duration` | mensal     |
| Fila (geral)            | DLQ < 0.1% dos jobs            | `bull_dlq_count`                   | mensal     |
| Backup PostgreSQL       | RPO ≤ 15min                    | PITR habilitado                    | contínua   |
| Recuperação de desastre | RTO ≤ 2h                       | Drill trimestral                   | trimestral |

**On-call:**

- Plantão em horário comercial no MVP.
- Escala 24/7 apenas quando SLA comercial justificar.

**Runbooks mínimos:**

- `docs/runbooks/identity-outage.md`
- `docs/runbooks/db-failover.md`
- `docs/runbooks/notification-backlog.md`
- `docs/runbooks/billing-webhook-replay.md`

## Consequências

**Positivas:**

- Compromisso mensurável e ajustável por SLA comercial.
- RPO/RTO explícitos informam custo de backup e arquitetura.
- Runbooks reduzem MTTR.

**Negativas:**

- 99.9% em monólito single-region pode exigir investimento em redundância.
- 24/7 com equipe pequena é caro.

**Mitigação:**

- Multi-AZ em produção para o banco (Neon oferece).
- API stateless em ≥ 2 instâncias atrás de load balancer.
- Plano de evolução para 24/7 atrelado ao crescimento de receita.
