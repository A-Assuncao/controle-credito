# ADR-0005: Autenticação e identidade

- **Status:** Refinado por ADR-0018
- **Data original:** 2026-06-20
- **Última atualização:** 2026-06-20

## Contexto

Necessário OIDC com capacidade de MFA (seção 20, NFR-03). Decisão entre provider gerenciado (Auth0, Clerk, Cognito) e self-hosted. O modelo de uso é **1 usuário por conta**, o que simplifica o desenho de identidade.

## Decisão

**Ory Kratos + Ory Hydra** (self-hosted, gratuitos) para identidade e OAuth2/OIDC.

- Kratos para cadastro, login, MFA TOTP, recuperação de conta.
- Hydra como Authorization Server para OIDC.
- Sessão única por conta (single-session): novo login revoga o anterior.
- MFA TOTP **opcional** por padrão; **obrigatório** para acesso a dados nominais PREMIUM (ADR-0018).
- Recuperação de senha por duplo fator (e-mail + celular) — ver ADR-0018.

## Consequências

**Positivas:**

- Sem custo de licença (importante: SAAS de baixo ticket para PF).
- Sem vendor lock-in.
- Código aberto, auditável.
- Controle total da trilha de auditoria.

**Negativas:**

- Mais um serviço para operar.
- Curva de configuração do Hydra (consent screen, JWKS).

**Mitigação:**

- Deploy via Docker Compose em dev e Helm chart em staging/prod.
- Testes de identidade em suíte dedicada.
- Runbook de identity-outage já redigido em `docs/runbooks/identity-outage.md`.

**Alternativa rejeitada:** Auth0/Clerk — custo recorrente elevado por usuário (especialmente em modelo PF).

**Refinamentos posteriores:** ver ADR-0018 (modelo single-user completo) e ADR-0019 (WhatsApp).
